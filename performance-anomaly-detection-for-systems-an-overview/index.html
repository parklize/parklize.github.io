Automated Performance Anomaly Detection for Systems: An Overview

Performance crisis occurs when availability or responsiveness goals are compromised by inevitable hardware and software problems [1]. Performance crises are detrimental in the context of data centers or telecommunication (telco) vendors who are serving a great number of people and services.

Desipite the fact that the operators in data centers or telco vendors have the highest priority to stabilize the system and avoid any performance anomaly, it is challenging due to the needs for inspecting a large number of collected system metrics (telemetry), logs, and alarms. Also, once an anomaly has been reported, it is crucial to quickly diagnose and fix that for business reasons. Therefore, many research works have been done to automate the (1) performance anomaly detection and (2) diagnosis in order to get rid of any anomaly as quickly as possible and also to shorten the diagnosis process of operators. Chen et al. [2] also refer to those two processes as (1) failure detection and (2) failure diagnosis.

Performance anomaly detection itself has been widely studied in the literature ([3],[4],[5] to name a few), which is not the focus in this post. Instead, this post aims to provide an overview for those automated diagnosis approaches for when a performance anomaly has been reported. As early as 2003, Joshua et al.[6] proposed the use of compute-intensive modeling techniques to automate such diagnose. Many researchers have followed up this direction and tried to identify operational problems by analyzing performance metrics using machine learning, which we classify into the following two categories:

  • "signatures" approach
  • non-"signatures" approach

"Signatures" Approach

This type of approaches construct "signatures" of performance anomalies based on historical data, e.g., performance metrics for different anomalies occured in the past at a data center. Once those signatures for each anomaly constructed, they can be used for diagnosing new problems by identifying which type of anomaly it belongs to based on its similarity with the signatures of previously-seen anomalies. Examples of this line of research include [7], [8], [9].

For example, HiLighter [7:1] proposed constructing singatures of a machine state by a subset of features selected via linear regression with L1 regularization. The same author [8:1] later proposed a updated "signatures" ("fingerprints" in the paper) approach for automatic classification and identification of crises/anomalies for servers in the context of data centers. This approach involves (1) automatically selecting a subset of relevant metrics out of all collected ones as HiLigter did, and (2) constructing a vector representation (signature) of each performance anomaly with quantile information of each selected metric.

Non-"Signatures" Approach

In contrast to the "signatures" approach, we can directly learn classifiers to classify given anomaly into previously-seen anomaly type(s) in the training data without relying on constructed signatures for each anomaly type.

Yuan et al. [10] proposed trace-based approach for diagnosing computer problems where a tracer will run and log the sequence of events (i.e., system calls on Windows XP) triggered in the system when the symptom of a problem is being reproduced. The sequence of events are represented with n-grams and mapped to a binary vector, which then is used as an input for SVM classifier.

Random Forest is one of the most popular models for detecting performance anomalies in the literature. In [11], the authors proposed online classification of faults for HPC systems where Random Forest outperforms other ML approaches such as Decision Tree, Support Vector Classifier and Feedforward-Neural Networks. The results show that almost perfect classification accuracy for different fault types based on Antarex dataset published along with this paper. Sauvanaud et al. [12] trained Random Forest models for each Virtual Network Function and for four different faults.
